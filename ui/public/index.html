About Safe Load Lab

Everything you need to know about load testing

🎯 What is Safe Load Lab?

Safe Load Lab is a controlled load testing tool designed for developers, QA engineers, and DevOps teams who want to test their own applications. Unlike cloud-based load testing services, it runs entirely on your machine with zero dependencies on external services, APIs, or subscriptions.

It sends simulated HTTP requests to your application to measure how it performs under load, revealing potential bottlenecks, capacity limits, and performance issues before they reach production.

📖 Understanding Load Testing Metrics

Total Requests: How many HTTP requests were sent to your application.

Success Rate: Percentage of requests that returned a successful response (2xx or 3xx). 98% = excellent, 80% = concerning, 50% = overloaded

Average Latency: Mean response time in milliseconds. 100ms = good, 500ms = slow, 2000ms+ = problem

P95 Latency: 95th percentile - 95% of requests were faster than this. Good for SLA planning.

P99 Latency: 99th percentile - Shows how your worst-case requests perform (almost always).

🔧 How to Use the Dashboard

Quick Mode (Recommended for Beginners)

Fill in simple fields:

  • Target URL - The endpoint to test (e.g., http://localhost:3000/api/health)
  • Duration - How long to run (start with 30-60 seconds)
  • Concurrency - Parallel requests (start with 5-10)
  • RPS - Requests per second (start with 10-25)

Advanced Mode (For Power Users)

Use full JSON configuration for staged testing, authentication, and custom headers:

{ "url": "http://localhost:3000/api/users", "stages": [ { "name": "warm-up", "durationSeconds": 30, "concurrency": 3, "rps": 8 }, { "name": "steady", "durationSeconds": 60, "concurrency": 25, "rps": 50 } ] }

⚠️ Safety Guidelines

Only test systems you own or have explicit permission to test.

  • Test your own local development environment
  • Test your company's staging/QA environments
  • Test production systems you own and maintain
  • Get written permission before testing shared infrastructure
  • Notify your team before running tests
  • Monitor your application during testing
  • Stop immediately if errors spike or the system becomes unresponsive

🛡️ Built-in Safety Limits

These hard limits protect against accidental resource exhaustion:

Maximum Duration: 600 seconds (10 minutes) Maximum Concurrency: 100 simultaneous requests Maximum RPS: 200 requests per second Maximum Body Size: 1 MB per request

All tests require confirmation that you own the target.

❓ Frequently Asked Questions

Do I need an API key or subscription?

No. Safe Load Lab runs entirely on your machine. No cloud service, LLM, or external API required.

Can I test authenticated endpoints?

Yes. Use Advanced mode to add authentication headers, cookies, or setup/login flows.

Where are my test reports saved?

All reports are saved in the reports/ directory.

Why are my results inconsistent?

Network latency, CPU usage, disk I/O, and other system factors vary. Run multiple tests to get better averages.

Can I test external websites?

Only if you own them. Load testing third-party sites without permission is illegal.

My test keeps failing with 4xx/5xx errors

Check: application URL, authentication headers, request body format, endpoint availability, server logs, and database connectivity.

🎓 Best Practices

  • Always validate your config before running
  • Start with low concurrency and RPS, then increase gradually
  • Monitor your application during tests (CPU, memory, database)
  • Run tests during off-peak hours on shared systems
  • Notify team members before running tests
  • Use staged testing for realistic patterns (warm-up → steady → cool-down)
  • Run the same test multiple times to verify consistency
  • Document your findings and track improvements over time

📞 Troubleshooting

Error: "Connection refused"

Your application is not running or the URL is wrong. Test it:

curl http://localhost:3000/api/health

Error: "sll: command not found"

The CLI tool isn't installed. From the project folder, run:

npm install -g .

Port 3001 already in use

Another process is using the port. Restart your computer or wait for the other process to finish.

Test won't start

Make sure all required fields are filled. URL must start with http:// or https://
